컴퓨팅이 "처리"를 담당한다면, 스토리지는 "보관"을 담당한다. 컴퓨팅 서비스를 둘러봤으니 이번에는 데이터를 어디에, 어떤 형태로 저장하는지를 살펴본다.

## 스토리지 서비스란

저장 방식은 크게 세 가지로 나뉘는데, 일상적인 비유로 구분하면 이해가 쉽다.

**객체 스토리지**는 사진첩과 비슷하다. 파일 하나하나를 통째로 저장하고 꺼내 쓴다. 파일 안의 일부만 수정하는 게 아니라 통째로 올리고 통째로 받는다.
**블록 스토리지**는 외장 하드와 비슷하다. 디스크처럼 작은 블록 단위로 데이터를 읽고 쓸 수 있어서, 파일의 일부분만 바꾸는 것도 가능하다. OS를 설치하고 데이터베이스를 돌리려면 이런 디스크가 필요하다.
**파일 스토리지**는 회사의 공유 폴더와 비슷하다. 여러 컴퓨터(서버)가 동시에 같은 폴더에 접근해서 파일을 읽고 쓸 수 있다.

## 핵심 개념과 구조

### 3분류와 AWS 서비스 매핑

![스토리지 3분류와 AWS 서비스](/assets/images/aws_storage_types.png)

**객체 스토리지 — Amazon S3**

S3는 파일을 "객체"로 저장하는 서비스로, 용량 제한이 거의 없고 가용성도 매우 높다. 같은 데이터라도 얼마나 자주 접근하느냐에 따라 비용을 줄일 수 있도록 여러 **스토리지 클래스**를 제공한다.

- **S3 Standard**: 자주 접근하는 데이터용. 비용은 높지만 즉시 조회 가능
- **S3 Standard-IA(Infrequent Access)**: 가끔 접근하지만 필요할 땐 즉시 꺼내야 하는 데이터용. 저장 비용은 낮지만 꺼낼 때(조회) 비용이 붙는다
- **S3 Glacier 계열**: 거의 접근하지 않는 보관용 데이터. 저장 비용은 가장 낮지만, 꺼내려면 몇 분~몇 시간이 걸린다

즉 "자주 쓰는 데이터일수록 비싸지만 빠르고, 안 쓰는 데이터일수록 싸지만 느리다"는 트레이드오프가 있다.

**블록 스토리지 — Amazon EBS / EC2 인스턴스 스토어**

- **Amazon EBS(Elastic Block Store)**: EC2에 붙여 쓰는 영구 디스크다. 인스턴스를 종료해도 EBS 볼륨은 따로 보존되고, 다른 인스턴스에 다시 연결할 수도 있다
- **EC2 인스턴스 스토어**: 물리 서버에 직접 붙어있는 임시 디스크다. 속도는 빠르지만 인스턴스를 멈추거나 종료하면 데이터가 사라진다. 캐시처럼 "없어져도 괜찮은" 임시 데이터에 적합하다

**파일 스토리지 — Amazon EFS / Amazon FSx**

- **Amazon EFS(Elastic File System)**: 여러 EC2 인스턴스가 동시에 마운트해서 쓸 수 있는 완전관리형 Linux용 파일 시스템
- **Amazon FSx**: Windows 파일 서버, Lustre(고성능 컴퓨팅용) 등 특정 워크로드에 최적화된 파일 시스템을 관리형으로 제공

**하이브리드 — AWS Storage Gateway**

온프레미스 환경과 클라우드 스토리지를 연결해주는 게이트웨이다. 사내 서버는 그대로 두고, 자주 쓰는 데이터는 로컬에 캐싱하면서 백업이나 오래된 데이터는 S3/Glacier로 자동 전송하는 식으로 점진적으로 클라우드를 활용할 수 있다.

**공통 운영 — 라이프사이클 정책과 백업**

- **라이프사이클 정책**: "30일 지난 객체는 IA로, 90일 지난 객체는 Glacier로" 같은 규칙을 정해두면 자동으로 더 저렴한 클래스로 옮겨준다. 사람이 일일이 옮길 필요가 없다
- **AWS Backup**: EBS, RDS, EFS 등 여러 서비스의 백업을 한곳에서 통합 관리하고 스케줄링한다

## 어디에 쓰이는가

가장 흔한 비용 최적화 사례는 로그 데이터다. 평소엔 거의 조회하지 않지만 규정상 일정 기간 보관해야 하는 애플리케이션 로그를 S3 Standard에 계속 쌓아두면 비용이 눈덩이처럼 늘어난다. 라이프사이클 정책으로 "30일 후 IA, 180일 후 Glacier"로 자동 이동시키면 같은 데이터를 훨씬 적은 비용으로 보관할 수 있다.
데이터베이스처럼 빈번한 읽기/쓰기가 필요한 워크로드는 EBS, 여러 웹서버가 같은 업로드 파일을 공유해야 하는 경우는 EFS를 쓰는 식으로 워크로드 특성에 맞춰 스토리지를 고른다.

## 직접 해보기 (AWS 콘솔)

S3 콘솔에서 버킷을 하나 만들고 파일을 업로드해보자. 업로드한 객체를 선택하면 스토리지 클래스를 Standard에서 다른 클래스로 바꿔볼 수 있는데, 이때 클래스별로 예상 비용과 조회 시간이 어떻게 달라지는지 안내 문구를 함께 확인해보면 좋다.

## 시험 포인트

- **S3(객체) vs EBS(블록) vs EFS(파일)**: 통째로 저장/조회하는 파일이면 S3, 디스크처럼 세밀한 읽기/쓰기가 필요하면 EBS, 여러 서버가 동시에 공유해야 하면 EFS
- **EBS vs 인스턴스 스토어**: EBS는 인스턴스 종료 후에도 데이터가 남는 영구 저장소, 인스턴스 스토어는 종료 시 데이터가 사라지는 임시 저장소
- **S3 스토리지 클래스**: Standard(빠름·비쌈) → IA(중간) → Glacier(느림·저렴) 순으로 접근 빈도가 낮아질수록 저장 비용은 내려가고 조회 시간/비용은 올라간다

## 정리

| 분류 | 비유 | AWS 서비스 |
|---|---|---|
| 객체 스토리지 | 사진첩 | Amazon S3 (Standard / IA / Glacier 등) |
| 블록 스토리지 | 외장 하드 | Amazon EBS(영구) / 인스턴스 스토어(임시) |
| 파일 스토리지 | 공유 폴더 | Amazon EFS(Linux) / Amazon FSx(Windows·Lustre 등) |
| 하이브리드 | 온프렘↔클라우드 연결 | AWS Storage Gateway |
| 공통 운영 | 자동 이동 / 백업 통합 | 라이프사이클 정책 / AWS Backup |
