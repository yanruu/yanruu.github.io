회귀분석 시리즈와는 별개로, 오늘부터 AWS 공인 자격증(AWS Certified Cloud Practitioner, CLF-C02) 준비 과정을 시리즈로 정리한다.
첫 글은 "AWS가 뭔지, 왜 쓰는지"부터 시작해서 시험 Domain 1(클라우드 개념, 전체 비중의 24%)에 해당하는 내용을 다룬다.

## AWS란 무엇인가

컴퓨터를 쓰려면 보통 서버, 저장장치, 네트워크 장비를 직접 사야 한다.
**클라우드 컴퓨팅**은 이런 장비를 직접 사지 않고, 인터넷을 통해 필요한 만큼만 빌려 쓰는 방식이다.

전기를 떠올리면 쉽다. 집에 발전기를 설치하는 대신 한전에서 전기를 끌어와 쓴 만큼만 요금을 내는 것처럼,
서버를 직접 구축하는 대신 AWS 같은 클라우드 사업자의 컴퓨팅 자원을 빌려서 쓴 만큼만 비용을 낸다.

**AWS(Amazon Web Services)**는 이런 클라우드 컴퓨팅 자원(컴퓨팅, 스토리지, 네트워킹, 데이터베이스 등)을
서비스 형태로 제공하는 사업자다. 이 시리즈에서는 AWS가 제공하는 서비스들을 자격증 시험 범위에 맞춰 하나씩 정리한다.

## 핵심 개념과 구조

### 왜 클라우드로 옮기는가 — 비용 구조의 변화

온프레미스(직접 서버를 사서 운영하는 방식)는 서버를 살 때 큰돈이 한 번에 들어간다.
이게 바로 **고정비(CapEx, Capital Expenditure)** 중심 구조다. 일단 사고 나면 트래픽이 많든 적든 비용은 거의 그대로다.

클라우드는 반대로 초기 투자가 거의 없고, **쓴 만큼만 내는 가변비(OpEx, Operating Expenditure)** 구조다.

![온프레미스 vs 클라우드 비용 구조](/assets/images/aws_cost_structure.png)

이 그래프에서 온프레미스는 초반에 큰 비용이 한 번에 발생한 뒤 완만하게 증가하지만,
클라우드는 시작 비용 없이 사용량에 비례해서 비용이 늘어나는 걸 확인할 수 있다.
이 구조 덕분에 AWS는 다음과 같은 장점을 갖는다.

- **고정비 → 가변비 전환**: 서버를 미리 사둘 필요가 없으니 초기 자본 부담이 줄어든다
- **규모의 경제(economies of scale)**: AWS는 전 세계 수많은 고객의 수요를 모아 대량으로 자원을 구매하기 때문에, 개별 기업이 직접 구축하는 것보다 단가가 낮다
- **용량을 추측할 필요 없음**: 트래픽이 갑자기 늘어도 자동으로 확장(Auto Scaling)하면 되니, "서버를 얼마나 사야 할지" 미리 추측해서 과다/과소 투자할 위험이 없다
- **속도와 민첩성**: 서버 구매·배송·설치에 걸리던 몇 주가 클릭 몇 번으로 몇 분이 된다
- **데이터센터 운영비 절감**: 전력, 냉각, 보안, 유지보수 인력에 들어가던 비용을 줄이고 핵심 비즈니스에 집중할 수 있다
- **글로벌 배포가 쉬움**: 전 세계 여러 지역에 클릭 몇 번으로 서비스를 배포할 수 있다

라이선스 비용 측면에서도 BYOL(Bring Your Own License, 기존에 사둔 라이선스를 그대로 가져와 쓰는 방식)과
AWS가 제공하는 라이선스 포함 요금제 중 선택할 수 있고,
이미 쓰고 있는 인스턴스 사양이 실제 필요량보다 큰 경우 사양을 낮추는 **rightsizing**도 클라우드 비용 최적화의 기본 동작이다.

### AWS 글로벌 인프라 — Region, Availability Zone, Edge Location

AWS는 전 세계 여러 지역에 데이터센터를 두고 있고, 이를 3단계 계층으로 나눈다.

![AWS 글로벌 인프라 구조](/assets/images/aws_global_infra.png)

이 그림에서 확인할 수 있듯 구조는 다음과 같다.

- **Region(리전)**: 서울, 도쿄, 버지니아처럼 지리적으로 구분된 큰 단위. 데이터 주권(어느 나라 법을 따르는지), 지연시간(사용자와 가까운 리전일수록 응답이 빠름) 때문에 리전을 선택하는 것이 중요하다
- **Availability Zone(AZ, 가용 영역)**: 하나의 리전 안에 물리적으로 분리된 데이터센터 묶음. 보통 한 리전에 AZ가 3개 이상 있고, 서로 저지연 전용망으로 연결되어 있다
- **Edge Location(엣지 로케이션)**: 사용자와 가장 가까운 곳에서 콘텐츠를 캐싱해 전달하는 위치 (CloudFront 같은 CDN 서비스가 사용)

여러 AZ에 서비스를 분산해두면(Multi-AZ) 한 AZ에 정전이나 장애가 나도 다른 AZ가 대신 처리할 수 있어 **고가용성(High Availability)**이 확보된다.
재해 복구나 법적 요건 때문에 아예 다른 리전에도 복제해두는 **Multi-Region** 구성을 쓰는 경우도 있다.

### AWS Well-Architected Framework — 잘 만든 아키텍처의 기준

클라우드에 뭔가를 만들 때 "이 정도면 잘 만든 건가?"를 판단할 기준이 필요하다.
AWS는 이를 6가지 기둥(pillar)으로 정리해뒀다.

![AWS Well-Architected Framework 6 pillars](/assets/images/aws_well_architected.png)

- **운영 우수성(Operational Excellence)**: 시스템을 운영·모니터링하고 지속적으로 개선하는 능력
- **보안(Security)**: 데이터와 시스템을 보호하는 능력 (다음 글에서 자세히 다룸)
- **안정성(Reliability)**: 장애에서 복구하고 수요 변화에 대응하는 능력
- **성능 효율성(Performance Efficiency)**: 컴퓨팅 자원을 효율적으로 사용하는 능력
- **비용 최적화(Cost Optimization)**: 불필요한 비용을 피하고 최저 비용으로 목표를 달성하는 능력
- **지속 가능성(Sustainability)**: 환경 영향을 최소화하는 능력 (최근 추가된 6번째 기둥)

이 6가지는 시험에도 자주 나오고, 실무에서 "이 설계가 괜찮은가?"를 점검할 때 실제로 쓰는 체크리스트이기도 하다.

### 클라우드로 옮기는 과정 — AWS CAF와 마이그레이션

기존 시스템을 클라우드로 옮기는 일은 단순히 서버를 복사하는 게 아니라 조직 전체의 변화다.
AWS는 이 변화를 체계적으로 준비하기 위한 가이드로 **AWS CAF(Cloud Adoption Framework)**를 제공한다.
비즈니스, 사람, 거버넌스, 플랫폼, 보안, 운영 6가지 관점에서 "우리 조직이 클라우드로 갈 준비가 됐는지"를 점검하는 틀이다.

실제 마이그레이션에는 대량의 데이터를 물리적 장치로 옮기는 **AWS Snowball**, 운영 중인 데이터베이스를 멈추지 않고 옮기는 **데이터베이스 복제** 같은 방법들이 쓰인다. (마이그레이션 서비스는 8번 글에서 더 다룬다.)

## 어디에 쓰이는가

스타트업이 처음부터 서버를 직접 구축하지 않고 AWS로 시작하는 이유, 대기업이 연말 세일처럼 트래픽이 몰리는 시기에만 서버를 늘렸다가 줄이는 이유가 모두 위에서 설명한 가변비 구조와 자동 확장 덕분이다.
반대로 보안·규제 때문에 일부 데이터만 사내에 두고 나머지는 클라우드에 두는 **하이브리드 클라우드**를 택하는 경우도 많다.

## 직접 해보기 (AWS 콘솔)

AWS 계정을 만들고 [AWS Management Console](https://console.aws.amazon.com)에 로그인하면 우측 상단에서 리전을 선택할 수 있다.
서울 리전(ap-northeast-2)으로 바꿔보고, 좌측 상단 검색창에 "EC2"를 입력해 어떤 인스턴스 옵션들이 있는지 한 번 둘러보자.
이번 글은 개념 위주라 직접 실행할 코드는 없고, 다음 글(보안과 IAM)부터 콘솔 실습이 본격적으로 나온다.

| AWS CAF | 클라우드 도입을 준비하는 조직 차원의 가이드 |
