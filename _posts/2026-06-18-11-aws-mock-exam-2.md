모의고사 1회에서 클라우드 개념과 보안 기초를 다뤘으니, 이번엔 시험에서 가장 비중이 큰(34%) **Domain 3: 클라우드 기술과 서비스** 범위다. 4~8번 글에서 다룬 컴퓨팅·스토리지·네트워킹·데이터베이스·AI/분석 서비스를 종합적으로 점검한다.

이 문제들도 1회와 마찬가지로 실제 시험 문제를 가져온 것이 아니라 학습 내용을 바탕으로 새로 구성한 오리지널 문제다.

## 문제

**Q1.** 한 회사가 컨테이너 기반 애플리케이션을 운영하려 한다. 컨테이너 오케스트레이션 표준인 쿠버네티스(Kubernetes)와 동일한 API와 도구를 그대로 쓰면서 AWS 위에서 관리형으로 운영하고 싶어한다. 가장 적합한 서비스는?

- **A.** Amazon ECS(자체 오케스트레이션 방식)
- **B.** Amazon EKS(Elastic Kubernetes Service)
- **C.** AWS Lambda
- **D.** Amazon Lightsail

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

Amazon EKS는 쿠버네티스를 관리형으로 실행해주는 서비스로, 쿠버네티스 표준 API와 생태계 도구를 그대로 쓸 수 있다. ECS는 AWS 자체 오케스트레이션 방식을 쓰는 컨테이너 서비스라 쿠버네티스 호환이 목적이라면 맞지 않다. Lambda는 컨테이너 오케스트레이션이 아닌 서버리스 함수 실행 서비스이고, Lightsail은 단순한 VPS형 서비스다.
</details>

---

**Q2.** 한 개발팀이 이벤트가 발생할 때만 짧게 실행되는 이미지 리사이징 함수를 만들려 한다. 서버를 미리 띄워둘 필요가 없고, 호출된 만큼만 비용을 내고 싶다. 가장 적합한 서비스는?

- **A.** Amazon EC2 On-Demand
- **B.** AWS Fargate
- **C.** AWS Lambda
- **D.** Amazon ECS(EC2 launch type)

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: C**

AWS Lambda는 이벤트 기반으로 짧게 실행되는 코드를 서버 관리 없이 실행하고, 실행된 시간만큼만 과금하는 서버리스 컴퓨팅 서비스다. Fargate도 서버를 직접 관리하지 않는다는 공통점이 있지만, 이는 컨테이너(긴 수명의 워크로드)를 서버리스로 운영하는 서비스라 "짧게 실행되는 단일 함수" 용도로는 Lambda가 더 적합하다. EC2 On-Demand와 ECS(EC2 launch type)는 모두 인스턴스를 직접 띄워두는 방식이라 이벤트 기반의 짧은 실행에는 비효율적이다.
</details>

**Q3.** 한 웹 서비스가 평소에는 트래픽이 적지만 특정 시간대에 갑자기 몰린다. 트래픽이 늘어나면 자동으로 인스턴스를 추가하고, 줄어들면 다시 줄이면서도, 여러 인스턴스에 요청을 고르게 분산하고 싶다. 이를 위한 조합으로 가장 적합한 것은?

- **A.** Auto Scaling 그룹 + Elastic Load Balancing(ELB)
- **B.** 단일 EC2 인스턴스 + 더 큰 인스턴스 타입으로 수동 교체
- **C.** Amazon S3 + CloudFront만 사용
- **D.** AWS Direct Connect로 온프레미스와 연결

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: A**

Auto Scaling 그룹은 트래픽(또는 지정한 지표)에 따라 인스턴스 수를 자동으로 늘리고 줄여주고, ELB는 여러 인스턴스에 요청을 분산해준다. 이 둘을 함께 쓰는 것이 가변적인 트래픽에 대응하는 전형적인 패턴이다. B는 수직 확장(scale-up)을 수동으로 하는 방식이라 자동화·분산과는 거리가 멀고, C와 D는 이 시나리오의 핵심 문제(트래픽 변화에 따른 컴퓨팅 자원 분산·확장)를 해결하지 못한다.
</details>

**Q4.** 한 회사가 6개월 이상 거의 접근하지 않는 로그 파일을 S3에 보관하면서 저장 비용을 최대한 낮추고 싶다. 다만 완전히 삭제하면 안 되고, 필요할 때(드물게) 몇 시간 내로 꺼낼 수 있어야 한다. 가장 적합한 접근 방식은?

- **A.** S3 Standard에 그대로 보관
- **B.** S3 Lifecycle 정책으로 일정 기간 후 S3 Glacier로 자동 전환
- **C.** 데이터를 EBS 볼륨으로 옮겨서 보관
- **D.** 매번 수동으로 파일을 다운로드해서 로컬에 보관

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

S3 Lifecycle 정책을 설정하면 일정 기간이 지난 객체를 자동으로 더 저렴한 스토리지 클래스(S3 Glacier 등)로 옮길 수 있다. Glacier는 저장 비용이 매우 낮은 대신 조회에 시간이 걸리는데, "드물게 접근하고 몇 시간 내 꺼낼 수 있으면 충분하다"는 조건과 잘 맞는다. A는 자주 접근하지 않는 데이터에 비용이 가장 비싼 클래스를 계속 쓰는 비효율적인 선택이고, C(EBS)는 특정 EC2 인스턴스에 연결되는 블록 스토리지라 이런 용도에 맞지 않으며, D는 자동화되지 않은 수동 방식이다.
</details>

**Q5.** EC2 인스턴스가 종료(stop/terminate)되면 함께 사라져도 괜찮은 임시 캐시 데이터를 저장하려 한다. 매우 높은 I/O 성능이 필요하지만 영속성은 중요하지 않다. 가장 적합한 스토리지는?

- **A.** Amazon EBS(범용 SSD)
- **B.** 인스턴스 스토어(Instance Store)
- **C.** Amazon S3
- **D.** Amazon EFS

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

인스턴스 스토어는 호스트 서버에 물리적으로 붙어 있는 임시 스토리지로, EBS보다 더 높은 I/O 성능을 내지만 인스턴스가 멈추거나 종료되면 데이터가 사라진다. "영속성은 중요하지 않고 성능이 중요한 임시 데이터"라는 조건이 인스턴스 스토어를 가리키는 전형적인 신호다. EBS는 인스턴스와 별개로 존재해 데이터가 유지되고, S3와 EFS는 객체/파일 스토리지로 이런 저지연 임시 캐시 용도에는 맞지 않는다.
</details>

**Q6.** 여러 대의 EC2 인스턴스가 동시에 같은 파일을 읽고 쓸 수 있는 공유 파일 시스템이 필요하다. 인스턴스 수가 늘어나도 용량이 자동으로 늘어나야 한다. 가장 적합한 서비스는?

- **A.** Amazon EBS
- **B.** 인스턴스 스토어
- **C.** Amazon EFS
- **D.** Amazon S3 Glacier

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: C**

Amazon EFS(Elastic File System)는 여러 EC2 인스턴스가 동시에 마운트해 공유할 수 있는 완전관리형 NFS 파일 시스템으로, 저장하는 만큼 자동으로 용량이 늘어난다. EBS는 기본적으로 한 번에 하나의 인스턴스에 연결되는 블록 스토리지라(일부 다중 연결 옵션은 있지만 일반적 용도는 단일 연결) 다수 인스턴스의 동시 공유에는 EFS가 표준적인 답이다. 인스턴스 스토어는 공유가 불가능한 로컬 스토리지이고, S3 Glacier는 파일 시스템이 아닌 아카이브용 객체 스토리지다.
</details>

**Q7.** 한 회사의 VPC 안에서, 서브넷 단위로 들어오고 나가는 트래픽을 무상태(stateless)로 허용/거부 규칙을 적용하고 싶다. 또한 인스턴스 단위로는 상태 저장(stateful) 방식으로 더 세밀하게 트래픽을 제어하고 싶다. 각각에 해당하는 서비스는?

- **A.** 서브넷 단위: 보안 그룹(Security Group) / 인스턴스 단위: 네트워크 ACL
- **B.** 서브넷 단위: 네트워크 ACL(NACL) / 인스턴스 단위: 보안 그룹(Security Group)
- **C.** 둘 다 보안 그룹으로만 처리 가능
- **D.** 둘 다 네트워크 ACL로만 처리 가능

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

네트워크 ACL은 서브넷 경계에서 동작하며 무상태(stateless)로, 들어오는 규칙과 나가는 규칙을 각각 명시해야 한다. 보안 그룹은 인스턴스(ENI) 단위에서 동작하며 상태 저장(stateful)으로, 들어오는 트래픽을 허용하면 그에 대한 응답은 자동으로 허용된다. 이 둘은 적용 범위(서브넷 vs 인스턴스)와 상태 저장 여부가 핵심 구분점이다.
</details>

**Q8.** VPC 안의 프라이빗 서브넷에 있는 EC2 인스턴스가 보안 패치를 받기 위해 인터넷으로 나가는 아웃바운드 연결은 필요하지만, 인터넷에서 그 인스턴스로 직접 들어오는 연결은 차단하고 싶다. 가장 적합한 구성은?

- **A.** 프라이빗 서브넷에 인터넷 게이트웨이를 직접 연결
- **B.** 퍼블릭 서브넷에 NAT 게이트웨이를 두고 프라이빗 서브넷의 라우팅을 그곳으로 연결
- **C.** 인스턴스에 퍼블릭 IP를 직접 할당
- **D.** Direct Connect로 온프레미스와 직접 연결

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

NAT 게이트웨이는 퍼블릭 서브넷에 배치하고, 프라이빗 서브넷의 라우팅 테이블이 NAT 게이트웨이를 거치도록 설정하면 프라이빗 인스턴스가 아웃바운드로는 인터넷에 나갈 수 있지만 인터넷에서 그 인스턴스로 직접 들어오는 연결은 막힌다. A는 프라이빗 서브넷의 정의(인터넷 게이트웨이로 직접 라우팅되지 않음)에 위배되고, C는 오히려 직접 노출을 만들며, D는 이 시나리오(인터넷 접근 제어)와 무관한 온프레미스 연결 수단이다.
</details>

**Q9.** 한 서비스가 읽기/쓰기 모두 매우 빠른 응답속도가 필요하고, 스키마가 자주 바뀌는 비정형 데이터를 다루며, 트래픽에 따라 자동으로 확장되는 완전관리형 NoSQL 데이터베이스를 원한다. 가장 적합한 서비스는?

- **A.** Amazon RDS(관계형 데이터베이스)
- **B.** Amazon Aurora
- **C.** Amazon DynamoDB
- **D.** Amazon Redshift

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: C**

DynamoDB는 완전관리형 NoSQL 키-값/문서 데이터베이스로, 스키마가 고정되지 않고 트래픽에 따라 자동으로 확장되며 한 자릿수 밀리초의 응답속도를 제공한다. RDS와 Aurora는 모두 관계형(고정 스키마, SQL) 데이터베이스라 "스키마가 자주 바뀌는 비정형 데이터"라는 조건과 맞지 않고, Redshift는 OLTP가 아닌 분석용 데이터 웨어하우스다.
</details>

**Q10.** 한 회사가 온프레미스에서 운영하던 오라클(Oracle) 데이터베이스를 Amazon Aurora로 마이그레이션하려 한다. 데이터베이스 엔진(스키마 구조 등)이 서로 다르기 때문에 스키마 변환 작업이 필요하고, 그 후 실제 데이터를 이전하면서 다운타임도 최소화하고 싶다. 이때 사용할 두 가지 서비스의 조합은?

- **A.** AWS Glue + Amazon Athena
- **B.** AWS Schema Conversion Tool(SCT) + AWS Database Migration Service(DMS)
- **C.** Amazon Kinesis + Amazon EMR
- **D.** Amazon Redshift + QuickSight

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

서로 다른 데이터베이스 엔진 간 마이그레이션에서는 먼저 AWS SCT로 스키마를 변환하고, 이후 AWS DMS로 실제 데이터를 이전한다(동종 엔진 간 마이그레이션이라면 SCT 없이 DMS만 쓰는 경우도 있다). A, C, D에 나열된 서비스들은 모두 ETL·스트리밍·분석·시각화에 쓰이는 서비스로 데이터베이스 엔진 마이그레이션 자체와는 목적이 다르다.
</details>

**Q11.** 한 회사가 이미 학습된 AI 모델을 API로 호출해서 문서 이미지에서 텍스트를 자동으로 추출하는 기능을 빠르게 추가하려 한다. 직접 모델을 만들 필요는 없다. 가장 적합한 서비스는?

- **A.** Amazon SageMaker AI
- **B.** Amazon Textract
- **C.** Amazon EMR
- **D.** AWS Glue

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

Amazon Textract는 문서 이미지에서 텍스트와 데이터를 추출해주는, 이미 학습된 AI 기능을 API로 제공하는 서비스다. SageMaker AI는 정반대로 처음부터 직접 모델을 구축·학습·배포하는 종합 플랫폼이라 "직접 모델을 만들 필요가 없다"는 조건과 맞지 않다. EMR과 Glue는 AI 모델 호출이 아니라 빅데이터 처리·ETL에 쓰이는 서비스다.
</details>

**Q12.** 한 시스템에서 주문이 들어오면 여러 백엔드 서비스(결제, 배송, 알림)가 그 주문을 순서대로 안정적으로 처리해야 한다. 처리하는 쪽이 잠시 멈춰도 메시지가 사라지지 않고 큐에 남아 있다가 다시 처리되어야 한다. 가장 적합한 서비스는?

- **A.** Amazon SNS
- **B.** Amazon SQS
- **C.** Amazon CloudFront
- **D.** AWS Direct Connect

<details markdown="1">

**정답: B**

Amazon SQS는 메시지를 큐에 쌓아두고 받는 쪽이 처리할 준비가 됐을 때 꺼내가는 메시지 큐 서비스로, 처리하는 쪽이 잠시 멈춰도 메시지가 큐에 남아 있다가 나중에 처리될 수 있다. SNS는 발행-구독 방식으로 메시지를 구독자들에게 즉시 뿌리는 데 적합하지만 "큐에 쌓아두고 순서대로 처리"라는 조건과는 SQS가 더 정확히 맞는다. CloudFront는 콘텐츠 전송(CDN), Direct Connect는 온프레미스-AWS 전용 네트워크 연결로 이 시나리오와 무관하다.
