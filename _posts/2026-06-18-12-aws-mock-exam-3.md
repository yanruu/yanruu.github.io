모의고사 1, 2회에서 Domain 1~3을 다뤘으니, 마지막 회차는 전체 9개 글을 종합한다. 특히 지금까지 비교적 가볍게 다뤘던 **Domain 4(청구·요금·지원, 12%)**에 더 비중을 두면서, Domain 1~3 내용도 다른 각도의 문제로 다시 점검한다.

이번에도 모든 문제는 실제 시험 문제가 아니라 학습 내용을 바탕으로 새로 구성한 오리지널 문제다.

## 문제

**Q1.** 한 회사가 클라우드로 전환하면서 "이제 서버 장비를 직접 구매하고 설치할 필요 없이, 클릭 몇 번으로 필요한 컴퓨팅 자원을 즉시 받을 수 있다"는 점을 가장 큰 장점으로 보고 있다. 이 장점에 해당하는 클라우드 컴퓨팅의 특징은?

- **A.** 다중 임차(Multi-tenancy)
- **B.** 주문형 셀프서비스(On-demand self-service)
- **C.** 규모의 경제(Economies of scale)
- **D.** 종량제 가격(Pay-as-you-go pricing)

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

사람의 개입(영업 담당자와의 협상, 장비 설치 등) 없이 필요한 자원을 즉시 받을 수 있다는 것이 주문형 셀프서비스의 정의다. D(종량제 가격)는 비용을 어떻게 청구하느냐에 대한 것이고, C(규모의 경제)는 AWS가 대규모로 자원을 운용해 단가를 낮추는 구조에 대한 것이라 이 문제의 핵심("즉시 받을 수 있다")과는 결이 다르다.
</details>

---

**Q2.** 한 회사가 신규 기능을 자주 배포해야 하는데, 배포 자동화나 장애 대응 절차가 표준화되어 있지 않아 배포마다 사고가 자주 발생한다. Well-Architected Framework의 어떤 기둥을 우선 개선해야 하는가?

- **A.** 보안(Security)
- **B.** 운영 우수성(Operational Excellence)
- **C.** 비용 최적화(Cost Optimization)
- **D.** 지속 가능성(Sustainability)

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

운영 우수성은 시스템을 운영·모니터링하고, 절차를 표준화하며, 작은 단위로 자주 안전하게 변경을 배포하는 능력에 초점을 둔다. "배포 자동화와 절차 표준화 부족"은 운영 우수성 기둥의 전형적인 개선 대상이다.
</details>

**Q3.** 한 회사가 Amazon S3에 저장된 고객 데이터의 암호화를 강화하려 한다. 공유 책임 모델에 따르면 S3에 저장된 데이터의 암호화 설정(예: 서버 측 암호화 활성화 여부)은 누구의 책임인가?

- **A.** AWS가 전적으로 책임진다
- **B.** 고객이 책임진다 — S3는 "클라우드 안의 보안"에 해당하는 설정을 고객이 구성해야 한다
- **C.** AWS와 고객이 책임지지 않고 제3자 감사기관이 책임진다
- **D.** S3는 관리형 서비스라 책임 분담 대상이 아니다

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

S3 같은 관리형 스토리지 서비스에서도 데이터 자체의 암호화 옵션을 켜고 끄는 것, 접근 정책을 설정하는 것은 고객의 책임이다. AWS는 S3 인프라 자체의 물리적·네트워크 보안("클라우드 자체의 보안")을 책임진다. 관리형 서비스라고 해서 고객 책임이 전혀 없는 것은 아니라는 점이 핵심이다.
</details>

**Q4.** 보안팀이 EC2 인스턴스에 설치된 소프트웨어 패키지에 알려진 CVE(취약점)가 있는지 자동으로 스캔하고 싶어한다. 가장 적합한 서비스는?

- **A.** Amazon GuardDuty
- **B.** Amazon Inspector
- **C.** Amazon Macie
- **D.** AWS Shield

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

Amazon Inspector는 EC2, 컨테이너, Lambda의 소프트웨어 취약점과 알려지지 않은 네트워크 노출을 자동으로 스캔하는 서비스다. GuardDuty는 위협 탐지(비정상 행동·악성 통신), Macie는 민감 데이터 탐지, Shield는 DDoS 방어를 담당해 역할이 서로 다르다.
</details>

**Q5.** 한 회사가 DDoS(분산 서비스 거부) 공격으로부터 인터넷에 노출된 웹 애플리케이션을 보호하고, 동시에 SQL 인젝션 같은 일반적인 웹 공격 패턴도 차단하고 싶어한다. 가장 적합한 조합은?

- **A.** AWS Shield(DDoS 방어) + AWS WAF(웹 공격 패턴 차단)
- **B.** Amazon GuardDuty 단독 사용
- **C.** AWS Config 단독 사용
- **D.** Amazon Macie + Amazon Inspector

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: A**

AWS Shield는 DDoS 공격을 방어하는 서비스이고, AWS WAF는 SQL 인젝션·XSS 같은 일반적인 웹 공격 패턴을 룰 기반으로 차단하는 웹 애플리케이션 방화벽이다. 두 서비스는 함께 쓰여 웹 애플리케이션을 다층으로 보호하는 전형적인 조합이다. GuardDuty는 위협 탐지(차단이 아닌 알림), Config는 설정 추적, Macie/Inspector는 각각 데이터·취약점 스캔으로 이 시나리오의 핵심(DDoS·웹 공격 차단)과는 맞지 않는다.
</details>

**Q6.** 트래픽이 예측 불가능한 새 마이크로서비스를 배포하려 한다. 서버 관리 부담 없이 코드 단위로 실행하고, 호출되지 않을 때는 비용이 전혀 들지 않아야 한다. 가장 적합한 서비스는?

- **A.** Amazon EC2 Reserved Instances
- **B.** AWS Lambda
- **C.** Amazon EC2 Dedicated Hosts
- **D.** AWS Outposts

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

AWS Lambda는 호출된 만큼만 과금되고 호출이 없으면 비용이 발생하지 않는 서버리스 컴퓨팅 서비스다. EC2 Reserved Instances와 Dedicated Hosts는 모두 약정·전용 자원을 미리 확보하는 방식이라 "호출 없을 때 비용 0"이라는 조건과 맞지 않고, Outposts는 온프레미스에 AWS 인프라를 확장하는 하이브리드 솔루션으로 이 시나리오와 무관하다.
</details>

**Q7.** 정적 웹사이트 파일(HTML, CSS, 이미지)을 저렴하게 호스팅하면서 전 세계 사용자에게 빠르게 전달하고 싶다. 가장 적합한 조합은?

- **A.** Amazon EC2 + Auto Scaling
- **B.** Amazon S3(정적 웹 호스팅) + Amazon CloudFront(CDN)
- **C.** Amazon RDS + Elastic Load Balancing
- **D.** AWS Direct Connect + VPN

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

정적 콘텐츠는 S3의 정적 웹 호스팅 기능으로 서버 없이 저장·서빙할 수 있고, CloudFront는 이 콘텐츠를 전 세계 엣지 로케이션에 캐싱해 사용자에게 빠르게 전달한다. A와 C는 서버를 띄우는 방식이라 정적 콘텐츠에는 과한 구성이고, D는 콘텐츠 전송과 무관한 전용 네트워크 연결 수단이다.
</details>

**Q8.** 자주 조회되지만 매번 데이터베이스까지 가서 조회하면 응답이 느려지는 데이터를 메모리에 캐싱해서 응답 속도를 크게 높이고 싶다. 가장 적합한 서비스는?

- **A.** Amazon ElastiCache
- **B.** Amazon S3 Glacier
- **C.** AWS Storage Gateway
- **D.** Amazon Redshift

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: A**

Amazon ElastiCache는 Redis 또는 Memcached 기반의 인메모리 캐싱 서비스로, 자주 조회되는 데이터를 메모리에 캐싱해 데이터베이스 조회 부담과 응답 시간을 크게 줄여준다. Glacier는 아카이브용 저속 스토리지, Storage Gateway는 온프레미스-클라우드 스토리지 연동, Redshift는 분석용 데이터 웨어하우스라 모두 이 시나리오(저지연 캐싱)와는 맞지 않는다.
</details>

**Q9.** S3에 누적된 원본 로그 데이터를 따로 옮기지 않고, 그 자리에서 SQL로 가끔 조회하고 싶다. 쿼리하지 않을 때는 비용이 들지 않아야 한다. 가장 적합한 서비스는?

- **A.** Amazon Redshift
- **B.** Amazon Athena
- **C.** Amazon RDS
- **D.** Amazon DynamoDB

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

Amazon Athena는 S3에 있는 데이터를 별도로 옮기지 않고 서버리스로 SQL 조회를 할 수 있는 서비스로, 쿼리한 만큼만 과금되어 쿼리하지 않을 때는 비용이 들지 않는다. Redshift는 데이터를 자체 클러스터로 적재해두고 정기적으로 분석하는 데 적합해 "그 자리에서 가끔 조회"라는 조건과는 맞지 않고, RDS와 DynamoDB는 OLTP용 데이터베이스로 이런 애드혹 분석 쿼리 용도가 아니다.
</details>

**Q10.** 한 스타트업이 신규 서비스를 막 시작해서 트래픽 패턴을 전혀 예측할 수 없다. 컴퓨팅 비용을 약정 없이 유연하게 운용하고 싶다. 트래픽 패턴이 안정되면 이후 비용 절감을 고려할 계획이다. 지금 시점에 가장 적합한 EC2 구매 옵션은?

- **A.** Reserved Instances(3년 약정)
- **B.** On-Demand
- **C.** Savings Plans(1년 약정)
- **D.** Spot Instances

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

On-Demand는 약정 없이 쓴 만큼 정가로 내는 방식으로, 트래픽을 예측할 수 없는 초기 단계에 가장 적합하다. A와 C는 모두 일정 기간 사용을 약정해야 하는데 지금처럼 사용량을 예측할 수 없는 단계에서는 위험하다. D(Spot)는 비용은 저렴하지만 AWS가 언제든 회수할 수 있어 안정적인 서비스 운영 초기 단계의 메인 워크로드에는 적합하지 않다(중단에 강한 보조 워크로드라면 고려할 수 있다).
</details>

**Q11.** 회사가 매달 비용이 예산을 초과하지 않는지 사전에 한도를 정해두고, 한도에 가까워지면 알림을 받고 싶어한다. 가장 적합한 도구는?

- **A.** AWS Cost Explorer
- **B.** AWS Budgets
- **C.** AWS Pricing Calculator
- **D.** AWS Trusted Advisor

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

AWS Budgets는 예산 한도를 설정하고, 실제 비용/사용량이 그 한도에 가까워지거나 초과하면 알림을 보내는 도구다. Cost Explorer는 과거 비용을 분석하는 사후 도구, Pricing Calculator는 사용 전 비용을 추정하는 사전 도구라 "한도 설정과 실시간 감시"라는 조건에는 Budgets가 정확히 맞는다.
</details>

**Q12.** 클라우드로 전환을 검토 중인 회사가 아직 아무 자원도 배포하지 않은 상태에서, EC2와 RDS를 특정 사양으로 구성했을 때 한 달에 예상 비용이 얼마나 나올지 미리 추정해보고 싶다. 가장 적합한 도구는?

- **A.** AWS Cost Explorer
- **B.** AWS Budgets
- **C.** AWS Pricing Calculator
- **D.** AWS Cost and Usage Report(CUR)

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: C**

AWS Pricing Calculator는 아직 사용하기 전에 구성을 입력하면 예상 비용을 추정해주는 사전 견적 도구다. Cost Explorer와 CUR은 이미 발생한 비용을 분석하는 사후 도구이고, Budgets는 한도 감시 도구라 "아직 배포 전, 비용을 미리 추정"이라는 조건에는 Pricing Calculator만 맞는다.
</details>

**Q13.** 한 회사가 프로덕션 환경에서 장애가 발생하면 30분 이내에 AWS의 답변을 받아야 하고, 전담 담당자가 계정 상황을 지속적으로 파악하고 있길 원한다(단, 가장 높은 등급의 15분 이내 응답까지는 필요하지 않다). 가장 적합한 Support 플랜은?

- **A.** Basic
- **B.** Developer
- **C.** Business
- **D.** Enterprise On-Ramp

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: D**

Enterprise On-Ramp는 긴급 건에 대해 30분 이내 응답을 제공하고 전담 TAM(Technical Account Manager) 풀에 접근할 수 있는 플랜이다. Enterprise(최상위 플랜)는 15분 이내 응답과 전담 TAM 1명 배정이 핵심인데, 문제에서 "15분 이내까지는 필요하지 않다"고 했으므로 그보다 한 단계 아래인 Enterprise On-Ramp가 가장 적합하다. Business는 24/7 기술 지원은 제공하지만 전담 TAM은 제공하지 않는다.
</details>

**Q14.** 여러 개의 AWS 계정을 운영하는 조직이 계정별로 따로 결제하지 않고 청구서를 하나로 통합하고, 계정들의 사용량을 합산해 더 큰 할인 혜택을 받고 싶어한다. 가장 적합한 기능은?

- **A.** IAM Identity Center
- **B.** AWS Organizations의 통합 결제(Consolidated Billing)
- **C.** Amazon Cognito
- **D.** AWS Trusted Advisor

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

AWS Organizations의 통합 결제 기능은 여러 계정을 하나의 조직으로 묶어 청구서를 한 번에 받고, 계정들의 사용량을 합산해 RI 할인 같은 볼륨 할인을 공유할 수 있게 해준다. IAM Identity Center는 SSO 접근 관리, Cognito는 애플리케이션 사용자 인증, Trusted Advisor는 계정 점검 도구로 모두 청구 통합과는 직접 관련이 없다.
</details>

**Q15.** 한 회사가 계정 내에서 사용하지 않는데 비용이 계속 나가는 자원(예: 연결되지 않은 EBS 볼륨, 유휴 EC2 인스턴스)을 찾아서 비용을 절감하고 싶어한다. 가장 적합한 도구는?

- **A.** AWS Health Dashboard
- **B.** AWS Trusted Advisor
- **C.** Amazon CloudWatch
- **D.** AWS CloudTrail

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

AWS Trusted Advisor는 비용, 보안, 성능, 내구성, 서비스 한도 다섯 영역에서 계정을 점검하는데, 비용 영역에서는 유휴 자원이나 연결되지 않은 볼륨처럼 "쓰지 않는데 돈이 나가는" 자원을 찾아 권고해준다. Health Dashboard는 AWS 측 이벤트(장애·유지보수) 알림, CloudWatch는 성능 모니터링, CloudTrail은 API 호출 기록이라 이 시나리오(불필요한 비용 자원 탐지)에는 Trusted Advisor가 정확히 맞는다.
</details>
