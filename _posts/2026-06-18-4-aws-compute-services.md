지금까지는 보안 관련 글 두 편이었다. 보안은 "어떻게 지키는가"의 영역이었고, 이제부터는 "AWS에서 실제로 코드를 어떻게 돌리는가"로 넘어간다.

## 컴퓨팅 서비스란

온프레미스에서는 코드를 돌리려면 서버를 사고, 운영체제를 설치하고, 패치하고, 트래픽이 늘면 서버를 더 사야 한다. 이 모든 걸 직접 관리해야 한다.

AWS의 컴퓨팅 서비스들은 "내가 얼마나 직접 관리할 것인가"의 스펙트럼으로 이해하면 쉽다. 한쪽 끝에는 가상 서버를 통째로 빌려주고 OS부터 직접 관리하게 하는 방식이 있고, 반대쪽 끝에는 서버라는 개념 자체를 숨기고 코드만 올리면 AWS가 알아서 실행해주는 방식이 있다. 그 사이에 컨테이너 기반 옵션들이 있다.

## 핵심 개념과 구조

### 관리 부담 스펙트럼 — EC2 → 컨테이너 → 서버리스

![컴퓨팅 서비스 - 관리 부담 스펙트럼](/assets/images/aws_compute_spectrum.png)

왼쪽일수록 내가 세밀하게 통제할 수 있지만 관리할 것도 많고, 오른쪽일수록 AWS가 알아서 해주지만 내가 통제할 수 있는 부분은 줄어든다. 공유 책임 모델 글에서 봤던 "관리형일수록 고객 책임이 줄어든다"는 원칙이 컴퓨팅 서비스에도 그대로 적용된다.

**Amazon EC2(Elastic Compute Cloud)**는 가상 서버를 빌려 쓰는 가장 기본적인 서비스다. 용도에 맞는 **인스턴스 유형**(범용, 컴퓨팅 최적화, 메모리 최적화 등)을 고르고, OS 설치부터 패치, 방화벽(보안 그룹) 설정까지 직접 한다. 구매 옵션(On-Demand/Reserved/Spot 등)에 따라 비용이 크게 달라지는데, 이 부분은 시험 비중이 커서 9번 글에서 따로 깊게 다룬다.

가운데에는 **컨테이너** 옵션들이 있다. 컨테이너는 애플리케이션과 실행 환경을 패키징해서 어디서든 똑같이 돌릴 수 있게 해주는 단위다.
- **Amazon ECS(Elastic Container Service)**: AWS 자체 컨테이너 오케스트레이션 서비스. 여러 컨테이너를 어디에 배치하고 몇 개 띄울지 관리해준다
- **Amazon EKS(Elastic Kubernetes Service)**: 오픈소스 표준인 **쿠버네티스(Kubernetes)**를 AWS가 관리해주는 버전. 이미 쿠버네티스를 쓰고 있거나 멀티클라우드를 고려한다면 EKS를 선택한다
- **Amazon ECR(Elastic Container Registry)**: 컨테이너 이미지를 저장하고 배포하는 저장소

가장 오른쪽에는 **서버리스(Serverless)**가 있다. 이름 그대로 서버 관리라는 개념이 사라진다.
- **AWS Lambda**: 코드를 올려두면 이벤트(요청, 파일 업로드 등)가 발생할 때만 실행되고, 실행된 시간만큼만 과금된다. 서버를 켜두고 기다릴 필요가 없다
- **AWS Fargate**: ECS/EKS에서 쓰는 컨테이너를 서버리스로 실행한다. 컨테이너는 쓰고 싶지만 EC2 인스턴스를 직접 관리하긴 싫을 때 쓴다. "Lambda는 함수 단위의 서버리스, Fargate는 컨테이너 단위의 서버리스"로 구분하면 된다

이 외에도 **AWS Batch**(대량 배치 작업 스케줄링), **AWS Elastic Beanstalk**(코드만 올리면 인프라를 자동 구성해주는 PaaS), **Amazon Lightsail**(소규모 프로젝트용 단순 가상서버+고정 요금제), **AWS Outposts**(AWS 인프라를 고객의 온프레미스 데이터센터에 그대로 가져다 놓는 서비스)가 있다.

### 배포/운영 방식

AWS 리소스를 다루는 방법은 크게 세 가지다. **콘솔(Console)**은 웹 브라우저로 클릭하며 다루는 방식, **API/SDK/CLI**는 코드나 명령줄로 다루는 방식, **IaC(Infrastructure as Code, 대표적으로 AWS CloudFormation)**는 인프라 구성을 코드(YAML/JSON)로 작성해두고 그대로 재현하는 방식이다. 실무에서는 반복 가능하고 버전 관리가 되는 IaC를 선호하지만, 학습 단계에서는 콘솔로 직접 클릭해보는 게 이해에 도움이 된다.

인프라를 어디에 둘지에 따라 **클라우드**(전부 AWS), **온프레미스**(전부 자체 데이터센터), **하이브리드**(일부는 AWS, 일부는 자체 인프라, Outposts 등으로 연결) 모델로 나뉜다.

### 트래픽 관리

- **Auto Scaling**: 트래픽이 늘면 인스턴스를 자동으로 늘리고, 줄면 자동으로 줄인다. 미리 용량을 추측해서 과다/과소 투자할 필요가 없다는 클라우드의 핵심 장점이 여기서 실현된다
- **ELB(Elastic Load Balancing)**: 여러 인스턴스에 트래픽을 고르게 분산한다. 한 인스턴스가 죽어도 ELB가 살아있는 인스턴스로만 트래픽을 보내기 때문에 가용성도 함께 올라간다

Auto Scaling과 ELB는 거의 항상 함께 쓰인다 — 인스턴스 수는 Auto Scaling이 조절하고, 그 인스턴스들로 트래픽을 나누는 건 ELB가 담당한다.

## 어디에 쓰이는가

트래픽 패턴이 어떤지에 따라 선택이 달라진다. 트래픽이 들쭉날쭉하고 호출 빈도가 낮은 API라면 쓴 만큼만 내는 **Lambda**가 유리하다. 이미 컨테이너로 패키징된 애플리케이션을 안정적으로 늘 띄워둬야 한다면 **ECS/EKS**가 적합하고, 특히 쿠버네티스 생태계를 이미 쓰고 있다면 EKS를 택한다. 세밀한 OS 설정이 필요하거나 특수한 소프트웨어를 깔아야 한다면 **EC2**가 답이다. 실무에서는 한 서비스 안에서도 이 셋을 섞어 쓰는 경우가 많다.

## 직접 해보기 (AWS 콘솔)

EC2 콘솔에서 프리 티어 대상 인스턴스(t2.micro 등)를 하나 띄워보고, 퍼블릭 IP로 접속해보자. 다 써본 뒤에는 반드시 인스턴스를 종료(Terminate)하거나 중지(Stop)해서 불필요한 과금을 막아야 한다.
Lambda 콘솔에서는 "Hello World" 같은 간단한 함수를 만들고 콘솔의 테스트 버튼으로 실행해보면, 서버를 띄우지 않고도 코드가 실행되는 걸 바로 체감할 수 있다.

## 시험 포인트

- **ECS vs EKS**: ECS는 AWS 자체 오케스트레이션(AWS 생태계에 최적화, 상대적으로 단순), EKS는 쿠버네티스 기반(표준 기술, 멀티클라우드 이전 용이). "AWS가 만든 것" vs "오픈소스 표준을 AWS가 관리해주는 것"으로 구분
- **Fargate vs Lambda**: 둘 다 서버리스지만, Fargate는 컨테이너를 서버리스로 실행(ECS/EKS와 함께 사용), Lambda는 함수(코드 조각) 단위로 실행. 실행 단위가 다르다
- **On-Demand vs Reserved vs Spot**: On-Demand는 약정 없이 쓴 만큼 정가로(트래픽이 예측 안 될 때), Reserved는 1·3년 약정으로 할인(꾸준히 쓸 게 확실할 때), Spot은 AWS의 유휴 자원을 싸게 쓰되 언제든 회수될 수 있음(중단돼도 괜찮은 배치 작업 등). 자세한 비교는 9번 글에서 다룬다

## 정리

| 구분 | 서비스 | 관리 부담 |
|---|---|---|
| 서버 | Amazon EC2 | 직접 관리 (OS, 패치, 방화벽 등) |
| 컨테이너 | ECS / EKS / ECR | AWS 또는 쿠버네티스가 오케스트레이션 관리 |
| 서버리스 | Lambda / Fargate | 서버 관리 없음 (함수 단위 / 컨테이너 단위) |
| 기타 | Batch / Elastic Beanstalk / Lightsail / Outposts | 배치 작업 / PaaS / 소규모 고정요금 / 온프렘 연동 |
| 트래픽 관리 | Auto Scaling / ELB | 자동 확장·축소 / 트래픽 분산 |
