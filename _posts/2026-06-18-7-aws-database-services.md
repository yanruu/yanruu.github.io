네트워킹으로 자원들이 어떻게 연결되는지를 봤다면, 이제 그 연결된 자원들 안에 결국 데이터를 어디에 저장하느냐를 봐야 한다. 스토리지 글에서 파일이나 디스크 단위 저장을 다뤘다면, 이번 글은 "구조화된 데이터를 검색하고 관리하는" 데이터베이스 영역이다.

## 데이터베이스 서비스란

데이터베이스도 직접 차를 수리하는 것과 정비소에 맡기는 것의 차이로 비교할 수 있다. EC2 인스턴스에 MySQL을 직접 설치하면, 설치부터 패치, 백업, 장애 복구, 확장까지 전부 내가 챙겨야 한다. 반면 **관리형 DB 서비스(RDS, DynamoDB 등)**를 쓰면 이런 운영 작업을 AWS가 대신 해주고, 나는 데이터와 쿼리에만 집중하면 된다.

이건 2번 글에서 다룬 공유 책임 모델과 그대로 이어진다. EC2에 직접 설치한 DB는 OS 패치부터 DB 엔진 패치까지 고객 책임이지만, RDS처럼 관리형 DB는 하드웨어, OS, DB 엔진 설치/패치, 백업 인프라까지 AWS가 책임지고 고객은 데이터와 접근 제어 정도만 신경 쓰면 된다.

## 핵심 개념과 구조

### 관계형 vs NoSQL vs 캐시

![데이터베이스 분류와 AWS 서비스](/assets/images/aws_db_classification.png)

**관계형 DB**는 엑셀 표처럼 정해진 스키마(열과 행의 구조)를 따르고, SQL로 데이터를 다룬다. 데이터 간의 관계(예: 주문-고객)를 엄격하게 관리해야 할 때 적합하다.

- **Amazon RDS**: MySQL, PostgreSQL, MariaDB, Oracle, SQL Server 같은 기존 DB 엔진을 관리형으로 제공한다. 기존에 쓰던 DB를 그대로 옮기고 싶을 때 적합하다
- **Amazon Aurora**: AWS가 자체 개발한 고성능 DB 엔진으로, MySQL·PostgreSQL과 호환되면서도 처리 속도와 가용성이 더 높다. 호환성이 우선이면 RDS, 성능과 가용성이 우선이면 Aurora를 선택한다

**NoSQL**은 유연한 문서함처럼 정해진 스키마 없이 데이터를 저장하고, 대규모로 확장하기 쉽다.

- **Amazon DynamoDB**: 키-값/문서형 NoSQL DB로, 완전 서버리스라서 직접 용량을 관리할 필요가 없고 트래픽에 따라 자동으로 확장된다
- 그 외 특수 목적 관리형 DB로 **Amazon DocumentDB**(MongoDB 호환 문서형 DB), **Amazon Neptune**(그래프 DB, 관계가 복잡하게 얽힌 데이터에 적합)이 있다

**캐시**는 단기 기억과 같다. 디스크가 아니라 메모리에 데이터를 올려두기 때문에 조회 속도가 훨씬 빠르다.

- **Amazon ElastiCache**: Redis 또는 Memcached 기반의 완전관리형 인메모리 캐시. 자주 조회되는 데이터를 메모리에 캐싱해두면, 매번 DB까지 가지 않고 빠르게 응답할 수 있어 DB 부하도 줄어든다

### 마이그레이션

기존에 운영 중인 DB를 AWS로 옮길 때 쓰는 도구다.

- **AWS DMS(Database Migration Service)**: DB의 데이터를 실제로 이전한다. 서비스를 멈추지 않고 운영 중인 DB를 그대로 복제하면서 마이그레이션하는 것도 가능하다
- **AWS SCT(Schema Conversion Tool)**: 서로 다른 DB 엔진 간에 **스키마(테이블 구조)**를 변환해준다. 예를 들어 Oracle 스키마를 PostgreSQL 스키마로 바꿔야 할 때 쓴다. DMS는 "데이터", SCT는 "구조"를 옮긴다고 구분하면 된다

## 어디에 쓰이는가

주문, 결제처럼 데이터 간의 관계가 명확하고 트랜잭션(여러 작업이 한 묶음으로 성공/실패해야 하는 처리)이 중요한 서비스는 **RDS/Aurora**가 적합하다. 사용자 수가 폭발적으로 늘 수 있는 모바일 앱의 세션 데이터나 IoT 센서 데이터처럼 단순한 키-값 조회가 대량으로 일어나는 경우엔 **DynamoDB**가 유리하다. 자주 조회되지만 자주 바뀌지 않는 데이터(인기 상품 목록, 세션 토큰)는 **ElastiCache**로 캐싱해서 DB 부하와 응답 시간을 동시에 줄인다.

## 직접 해보기 (AWS 콘솔)

RDS 콘솔에서 프리 티어 대상 인스턴스(db.t2.micro 등)로 MySQL 데이터베이스를 하나 만들어보자. 생성 후 보안 그룹에서 본인 IP만 접근을 허용하도록 설정하는 것도 함께 연습하면 좋다.
DynamoDB 콘솔에서는 테이블을 하나 만들고 항목(item)을 몇 개 직접 입력해보면, 스키마를 미리 정의하지 않고도 데이터를 바로 저장할 수 있다는 NoSQL의 특징을 체감할 수 있다.

## 시험 포인트

- **RDS vs Aurora**: RDS는 기존 DB 엔진과의 호환성 중심, Aurora는 MySQL/PostgreSQL 호환이면서 더 높은 성능과 가용성 중심
- **관계형(RDS) vs NoSQL(DynamoDB)**: 정해진 스키마와 복잡한 관계, 트랜잭션이 중요하면 관계형. 스키마가 유동적이고 대규모 확장이 우선이면 NoSQL
- **DMS vs SCT**: DMS는 데이터 자체를 이전, SCT는 서로 다른 DB 엔진 간 스키마(구조)를 변환. 보통 SCT로 스키마를 바꾸고 DMS로 데이터를 옮기는 순서로 함께 쓰인다

## 정리

| 분류 | 비유 | AWS 서비스 |
|---|---|---|
| 관계형 DB | 엑셀 표 (정해진 스키마, SQL) | Amazon RDS / Amazon Aurora |
| NoSQL | 유연한 문서함 (스키마 없음) | Amazon DynamoDB (+ DocumentDB, Neptune) |
| 캐시 | 단기 기억 (메모리 기반) | Amazon ElastiCache |
| 마이그레이션 | 데이터 이전 / 스키마 변환 | AWS DMS / AWS SCT |
