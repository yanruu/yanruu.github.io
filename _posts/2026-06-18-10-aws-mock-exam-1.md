학습 시리즈 9편을 모두 마쳤으니, 이제 직접 만든 모의고사로 점검해본다. 이 모의고사는 AWS 공식 시험 문제를 그대로 가져온 것이 아니라, 학습 시리즈에서 다룬 개념을 바탕으로 직접 구성한 **오리지널 문제**다. AWS 자격증 응시 동의서에는 시험 문제를 외부에 공개하지 않겠다는 비공개 조항이 있어서, 실제 문항을 재현하는 것은 금지되어 있다. 그래서 같은 개념을 묻더라도 상황과 표현은 전부 새로 만들었다.

총 3회로 구성된 모의고사 시리즈 중 첫 회로, **1~3번 글(Domain 1: 클라우드 개념, Domain 2: 보안과 컴플라이언스)** 범위를 다룬다. 전체 시리즈가 시험 도메인을 어떻게 나눠 커버하는지는 아래 그림을 참고하자.

![AWS CCP 도메인별 출제 비중과 모의고사 커버리지](/assets/images/aws_mock_exam_coverage.png)

문제마다 선택지 아래 "정답 및 해설 보기"를 클릭하면 정답과 설명이 펼쳐진다. 먼저 직접 풀어본 뒤에 펼쳐보는 걸 추천한다.

## 문제

**Q1.** 한 스타트업이 막 서비스를 시작하는 단계라 트래픽이 거의 없지만, 갑자기 입소문이 나면 트래픽이 폭증할 수도 있는 상황이다. 초기 서버 구매 비용을 들이지 않고, 필요할 때만 자원을 늘리고 줄이고 싶다. 이런 요구에 가장 부합하는 클라우드 컴퓨팅의 특징은?

- **A.** 온프레미스 데이터센터의 고정 용량 설계
- **B.** 주문형 셀프서비스(On-demand self-service)와 신속한 탄력성(Rapid elasticity)
- **C.** 특정 하드웨어 제조사와의 장기 유지보수 계약
- **D.** 데이터센터를 직접 소유하고 운영하는 자본 투자(CapEx) 모델

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

클라우드 컴퓨팅의 핵심 특징 중 "주문형 셀프서비스"는 사람이 개입하지 않고 필요할 때 즉시 자원을 받을 수 있다는 뜻이고, "신속한 탄력성"은 트래픽에 맞춰 자원을 빠르게 늘리고 줄일 수 있다는 뜻이다. A와 D는 온프레미스의 특징이고, C는 클라우드와 무관한 전통적 IT 계약 방식이다.
</details>

---

**Q2.** 한 회사가 자체 데이터센터를 운영하던 시절에는 서버를 구매하는 데 큰 초기 비용(자본 지출)을 들였다. 클라우드로 전환한 뒤에는 실제로 사용한 만큼만 매달 비용을 지불하는 방식으로 바뀌었다. 이런 비용 구조의 변화를 가장 정확히 설명하는 것은?

- **A.** OpEx(운영 지출)에서 CapEx(자본 지출)로 전환
- **B.** CapEx(자본 지출)에서 OpEx(운영 지출)로 전환
- **C.** 비용 구조에는 변화가 없고 결제 주기만 바뀜
- **D.** 클라우드는 항상 CapEx와 OpEx를 동시에 요구함

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

직접 서버를 사두는 것은 미래 사용을 위해 큰돈을 미리 투자하는 자본 지출(CapEx)이다. 클라우드의 "쓴 만큼 낸다" 모델은 매달 실제 사용량에 따라 비용을 치르는 운영 지출(OpEx)에 해당한다. 클라우드 경제성의 핵심 논리 중 하나가 바로 이 CapEx → OpEx 전환이다.
</details>

---

**Q3.** 한 글로벌 서비스가 아시아와 유럽 사용자에게 모두 낮은 지연시간(latency)으로 응답하고자 한다. 동시에 한 지역에 장애가 발생해도 다른 지역에서 서비스를 계속 제공할 수 있어야 한다. 이를 위해 활용해야 할 AWS 글로벌 인프라 개념은?

- **A.** 하나의 가용 영역(AZ) 안에 모든 서버를 배치
- **B.** 여러 리전(Region)에 걸쳐 서비스를 배포
- **C.** 하나의 엣지 로케이션에만 콘텐츠를 캐싱
- **D.** 단일 데이터센터의 서버 대수를 최대한 늘림

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

리전은 지리적으로 떨어진 독립된 위치다. 아시아와 유럽처럼 멀리 떨어진 사용자에게 낮은 지연시간을 주려면 사용자와 가까운 리전에 각각 서비스를 배포해야 한다. 또한 여러 리전에 걸쳐 배포하면 한 리전에 장애가 생겨도 다른 리전이 서비스를 이어갈 수 있다. A와 D는 가용성을 높이는 방법이 아니라 단일 지점에 집중하는 방식이라 오히려 위험하고, C(엣지 로케이션)는 콘텐츠 캐싱을 통한 지연시간 단축에는 도움이 되지만 장애 대응 자체를 해결하지는 않는다.
</details>

**Q4.** 한 회사가 시스템을 설계할 때 "정기적으로 백업을 점검하고, 장애 복구 절차를 문서화하며, 시스템 일부가 죽어도 전체 서비스가 멈추지 않게 한다"는 원칙을 최우선으로 삼고 있다. AWS Well-Architected Framework의 어떤 기둥(pillar)에 가장 가까운 우선순위인가?

- **A.** 비용 최적화(Cost Optimization)
- **B.** 성능 효율성(Performance Efficiency)
- **C.** 안정성(Reliability)
- **D.** 운영 우수성(Operational Excellence)

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: C**

장애로부터 복구하고, 일부 구성 요소의 실패에도 전체 시스템이 계속 동작하게 만드는 것은 안정성(Reliability) 기둥의 핵심이다. 운영 우수성은 시스템을 운영·모니터링·개선하는 프로세스 자체에 초점을 두고, 안정성은 "장애가 나도 서비스가 살아있는가"에 초점을 둔다는 차이를 기억하자.
</details>

---

**Q5.** 한 조직이 클라우드로 전환하면서 단순히 기술 이전뿐 아니라, 조직 구조 변화·직원 교육·새로운 거버넌스 정책까지 함께 고려하는 종합적인 전환 전략을 세우고 싶어한다. 이런 조직 차원의 클라우드 도입 전략을 체계적으로 정리해주는 AWS의 프레임워크는?

- **A.** AWS Well-Architected Framework
- **B.** AWS Cloud Adoption Framework(CAF)
- **C.** AWS Shared Responsibility Model
- **D.** AWS Trusted Advisor

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

Well-Architected Framework는 개별 워크로드(시스템)의 아키텍처 설계 품질에 초점을 둔다. 반면 AWS CAF는 비즈니스·사람·거버넌스·플랫폼·보안·운영 등 조직 전체의 클라우드 전환 전략을 다루는 더 상위 차원의 프레임워크다. 문제에서 "조직 구조 변화", "직원 교육", "거버넌스 정책"처럼 기술을 넘어선 조직적 요소를 언급한 것이 CAF를 가리키는 핵심 신호다.
</details>

**Q6.** 한 회사가 AWS 클라우드에서 EC2 인스턴스를 운영 중이다. 이 인스턴스에 설치된 운영체제의 보안 패치를 누가 책임지는지 궁금해졌다. 공유 책임 모델(Shared Responsibility Model)에 따르면 옳은 설명은?

- **A.** AWS가 게스트 OS 패치까지 전부 책임진다
- **B.** 고객이 게스트 OS 패치, 애플리케이션 보안 설정을 책임진다
- **C.** 물리적 데이터센터 보안과 게스트 OS 패치 모두 고객 책임이다
- **D.** EC2는 책임 분담 대상이 아니라 AWS가 전적으로 관리하는 관리형 서비스다

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

EC2처럼 고객이 직접 제어하는(unmanaged) 서비스에서는 "클라우드 안의 보안(Security IN the cloud)"을 고객이 책임진다. 여기에는 게스트 OS 패치, 방화벽 설정, 애플리케이션 보안, 데이터 암호화 설정 등이 포함된다. AWS는 물리적 시설, 하드웨어, 네트워크 인프라, 가상화 계층처럼 "클라우드 자체의 보안(Security OF the cloud)"을 책임진다. EC2는 고객이 OS 단까지 직접 관리하는 대표적인 unmanaged 서비스라 책임 분담의 전형적인 예시로 자주 나온다.
</details>

**Q7.** 한 관리자가 AWS 계정을 처음 만들고 나서, 일상적인 작업에는 루트 사용자를 쓰지 않고 별도의 IAM 사용자를 만들어 쓰기로 했다. 이 결정과 함께 적용해야 할 가장 중요한 보안 모범 사례는?

- **A.** 루트 사용자의 액세스 키를 코드에 하드코딩해서 자동화에 활용한다
- **B.** 루트 사용자와 모든 IAM 사용자에 다중 인증(MFA)을 활성화한다
- **C.** 모든 IAM 사용자에게 AdministratorAccess 정책을 기본으로 부여한다
- **D.** 비밀번호 정책을 비활성화해 로그인을 더 간편하게 만든다

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

루트 사용자는 계정 전체에 대한 무제한 권한을 가지므로 일상 작업에 쓰지 않는 것이 첫 번째 모범 사례이고, 루트 사용자와 IAM 사용자 모두에 MFA를 활성화하는 것이 두 번째 핵심 사례다. A는 자격 증명을 코드에 노출하는 대표적인 안티패턴이고, C는 최소 권한 원칙(필요한 권한만 부여)에 정반대되는 선택이며, D는 보안을 오히려 약화시킨다.
</details>

**Q8.** 여러 개의 AWS 계정을 운영하는 회사가 있다. 직원들이 각 계정마다 별도의 IAM 사용자와 비밀번호를 만들지 않고, 사내 디렉터리(예: 사내 SSO)로 한 번 로그인하면 권한이 있는 모든 AWS 계정에 접근할 수 있게 하고 싶다. 이런 요구에 가장 적합한 서비스는?

- **A.** IAM 사용자(IAM User)를 계정마다 개별 생성
- **B.** AWS IAM Identity Center(옛 AWS SSO)
- **C.** 루트 사용자 자격 증명 공유
- **D.** Amazon Cognito 사용자 풀

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

AWS IAM Identity Center는 여러 AWS 계정과 애플리케이션에 대한 SSO(Single Sign-On) 접근을 중앙에서 관리하는 서비스로, 사내 디렉터리와 연동해 한 번의 로그인으로 여러 계정에 접근할 수 있게 해준다. A는 계정마다 사용자를 따로 만드는 비효율적인 방식이고, C는 명백한 보안 위반이다. D의 Cognito는 자체적으로 만든 애플리케이션의 최종 사용자(고객) 인증에 쓰이는 서비스로, 직원의 AWS 계정 접근 관리와는 용도가 다르다.
</details>

**Q9.** 보안팀이 AWS 환경 전반에서 비정상적인 API 호출, 알려진 악성 IP와의 통신, 손상 가능성이 있는 자격 증명 사용 같은 위협을 자동으로 탐지하고 싶어한다. 머신러닝 기반으로 이런 위협을 지속적으로 모니터링해주는 서비스는?

- **A.** Amazon Inspector
- **B.** Amazon Macie
- **C.** Amazon GuardDuty
- **D.** AWS Config

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: C**

Amazon GuardDuty는 CloudTrail, VPC Flow Logs, DNS 로그 등을 머신러닝으로 분석해 비정상적인 활동이나 알려진 위협 패턴을 탐지하는 위협 탐지 서비스다. Inspector는 EC2·컨테이너·Lambda의 소프트웨어 취약점을 스캔하는 서비스이고, Macie는 S3에 저장된 데이터에서 개인식별정보(PII) 같은 민감 데이터를 찾아주는 서비스다. Config는 위협 탐지가 아니라 리소스 설정 변경을 추적하고 규정 준수 여부를 평가하는 서비스다.
</details>

**Q10.** 한 회사가 "지난주에 누가 이 S3 버킷의 정책을 변경했는지", "그 변경이 언제 일어났는지"를 추적하고 싶어한다. 계정 내 모든 API 호출 기록(누가, 언제, 무엇을)을 남겨주는 서비스는?

- **A.** Amazon CloudWatch
- **B.** AWS CloudTrail
- **C.** AWS Config
- **D.** AWS Trusted Advisor

<details markdown="1">
<summary>정답 및 해설 보기</summary>

**정답: B**

AWS CloudTrail은 계정 내에서 발생한 모든 API 호출(누가, 언제, 어떤 작업을)을 기록하는 감사 로그 서비스다. "누가 무엇을 변경했는지"를 추적하는 질문은 CloudTrail을 가리키는 전형적인 신호다. CloudWatch는 성능 지표·로그·알람 중심의 모니터링 서비스이고, Config는 리소스의 "현재/과거 설정 상태"를 추적하고 규정 준수를 평가하는 서비스로, CloudTrail의 "누가 했는가"와는 초점이 다르다.
</details>
