이전 글에서는 AWS가 무엇이고 왜 쓰는지를 다뤘다. 이번 글부터는 시험 비중이 가장 큰 Domain 2(보안과 컴플라이언스, 30%)를 다룬다.
그 중 가장 먼저 알아야 할 건 "보안에 대해 AWS와 내가 각각 어디까지 책임지는가"를 정하는 공유 책임 모델(Shared Responsibility Model)이다.

## 공유 책임 모델이란

건물을 빌려 쓴다고 생각해보자. 건물주는 건물 자체(구조, 전기 배선, 소방 시설)를 책임지고, 입주자는 자기 사무실 안의 물건(가구, 잠금장치, 출입 카드 관리)을 책임진다.
둘 다 안전을 위해 각자 역할이 있고, 어느 한쪽만 잘한다고 전체가 안전해지지는 않는다.

AWS도 마찬가지다. AWS는 클라우드 자체의 보안(**Security OF the Cloud**)을 책임지고, 고객은 클라우드 안에서 무엇을 어떻게 설정하느냐의 보안(**Security IN the Cloud**)을 책임진다.
이 경계를 정확히 아는 게 시험에서도, 실무에서도 가장 중요한 출발점이다.

## 핵심 개념과 구조

### 책임의 경계 — Security OF the Cloud vs Security IN the Cloud

![공유 책임 모델 — AWS 책임 vs 고객 책임](/assets/images/aws_shared_responsibility.png)

이 그림처럼 책임은 위아래로 나뉜다. 아래쪽(AWS 책임)은 데이터센터, 하드웨어, 네트워크 인프라, Region/AZ/Edge Location 같은 글로벌 인프라처럼 "클라우드 자체"를 이루는 부분이다.
위쪽(고객 책임)은 그 위에 올려놓은 내 데이터, 내가 설정한 IAM 권한, 내가 고른 OS와 방화벽 설정, 암호화 여부처럼 "내가 클라우드를 어떻게 쓰는지"에 해당하는 부분이다.

다만 이 경계는 서비스마다 위치가 달라진다. AWS가 더 많은 부분을 관리해주는 **관리형 서비스**일수록 고객이 직접 신경 써야 할 범위는 줄어든다.

| 서비스 | AWS가 관리 | 고객이 관리 |
|---|---|---|
| EC2 | 하드웨어, 가상화 계층, 네트워크 인프라 | OS, 미들웨어, 애플리케이션, OS 보안 패치, 보안 그룹(방화벽) 설정, 데이터 |
| RDS | 하드웨어, OS, DB 엔진 설치/패치, 백업 인프라 | 데이터, 접근 제어(IAM·보안 그룹), 일부 DB 파라미터 설정 |
| Lambda | 하드웨어, OS, 런타임, 자동 스케일링 | 코드, 코드 안의 데이터 처리 로직, IAM 실행 권한 |

EC2 → RDS → Lambda 순으로 AWS가 관리해주는 영역이 넓어지고, 그만큼 고객이 직접 패치하거나 설정할 부분은 줄어든다.
대신 고객이 세밀하게 통제할 수 있는 부분도 함께 줄어든다는 트레이드오프가 있다.

### IAM — 누가 무엇을 할 수 있는지 관리하기

**IAM(Identity and Access Management)**은 "누가(Who) AWS 안에서 무엇을(What) 할 수 있는지"를 관리하는 서비스다. 4가지 구성요소로 이루어진다.

- **사용자(User)**: 실제 사람이나 애플리케이션을 나타내는 개별 자격증명
- **그룹(Group)**: 사용자들을 모아놓은 묶음. 그룹에 정책을 붙이면 그룹 내 모든 사용자에게 한꺼번에 적용된다
- **역할(Role)**: 특정 권한을 "빌려 쓰는" 임시 자격증명. 사람이 아니라 EC2 인스턴스나 다른 AWS 계정이 맡아서 쓸 수 있다 (예: EC2가 역할을 맡아서 S3에 접근)
- **정책(Policy)**: "누가 무엇을 할 수 있는지/없는지"를 적어둔 JSON 문서

계정을 처음 만들 때 생기는 **root 계정**은 모든 권한을 가지고 있어서, 일상 업무에는 절대 쓰지 않고 MFA로 잠가두는 게 원칙이다. 결제 정보 변경처럼 root만 할 수 있는 작업이 아니면 IAM 사용자나 역할로 작업한다.

권한을 줄 때는 **최소 권한 원칙(least privilege)**을 따른다. "나중에 필요할 수도 있으니" 미리 넓은 권한을 주지 않고, 딱 필요한 만큼만 주고 필요해지면 그때 추가한다.

### 자격증명을 더 안전하게 관리하는 도구들

- **액세스 키(Access Key)**: 프로그래밍 방식(API·CLI)으로 AWS를 호출할 때 쓰는 키. 콘솔 로그인은 비밀번호로 충분하지만, 코드에서 AWS를 호출할 땐 액세스 키나 역할이 필요하다
- **비밀번호 정책**: 최소 길이, 특수문자 포함 여부, 만료 주기 같은 규칙을 계정 전체에 강제할 수 있다
- **MFA(다중 인증)**: 비밀번호 외에 OTP 앱이나 보안 키 같은 추가 인증 수단을 요구한다. root 계정과 권한이 큰 사용자에게는 필수로 걸어야 한다
- **AWS Secrets Manager**: DB 비밀번호, API 키처럼 민감한 자격증명을 코드에 하드코딩하지 않고 안전하게 저장하고, 주기적으로 자동 교체(rotation)해주는 서비스
- **AWS Systems Manager**: 여러 EC2 인스턴스에 패치를 일괄 적용하거나, 설정값(파라미터)을 중앙에서 관리하는 운영 도구
- **AWS IAM Identity Center**(구 AWS SSO): 여러 AWS 계정과 외부 애플리케이션에 한 번의 로그인으로 접근하게 해주는 서비스. 회사에 AWS 계정이 여러 개 있을 때 유용하다
- **크로스 계정 역할(Cross-account Role)**: 다른 AWS 계정의 사용자가 내 계정의 특정 역할을 빌려서 권한을 행사하게 하는 방식. 개발 계정/운영 계정처럼 계정을 나눠 쓸 때 자주 쓴다
- **페더레이션 ID(Federated Identity)**: 회사 사내 계정(Active Directory), SAML 기반 SSO, 소셜 로그인 등 외부 자격증명으로 AWS에 접근하는 방식. 사용자마다 별도 IAM 계정을 만들지 않아도 된다

## 어디에 쓰이는가

실무에서는 사람마다 IAM 사용자를 따로 만들기보다, "개발팀", "운영팀" 같은 그룹을 만들어 정책을 그룹에 붙이고 사람을 그룹에 넣고 빼는 방식으로 관리한다.
root 계정은 결제 정보 변경처럼 꼭 필요한 경우에만 MFA를 건 상태로 잠깐 쓰고 다시 잠가둔다.

## 직접 해보기 (AWS 콘솔)

IAM 콘솔에서 사용자를 만들고, "개발자" 같은 그룹을 만들어 `AmazonS3ReadOnlyAccess` 같은 AWS 관리형 정책을 붙여보자.
그 다음 본인 계정에 MFA 디바이스(Google Authenticator 같은 OTP 앱)를 등록해보면, 로그인할 때 인증 단계가 어떻게 추가되는지 직접 체감할 수 있다.

## 시험 포인트

- **"고객 책임" vs "AWS 책임"**: 서비스가 관리형일수록(Lambda > RDS > EC2) AWS 책임 범위가 커진다. 보안 패치는 EC2에서는 고객 책임, RDS·Lambda에서는 AWS 책임
- **root 계정 vs IAM 사용자**: root는 모든 권한을 가지며 일상 업무에 쓰지 않는다. IAM 사용자는 필요한 권한만 제한적으로 부여받는다
- **IAM 정책 vs 리소스 기반 정책**: IAM 정책은 사용자·그룹·역할에 붙고, S3 버킷 정책 같은 리소스 기반 정책은 리소스 자체에 붙는다. 둘 다 JSON 문서지만 "누구에게 붙는지"가 다르다

## 정리

| 개념 | 설명 |
|------|------|
| 공유 책임 모델 | AWS는 Security OF the Cloud, 고객은 Security IN the Cloud를 책임 |
| IAM 사용자/그룹/역할/정책 | 각각 개별 자격증명 / 사용자 묶음 / 임시로 빌려 쓰는 권한 / 권한을 적은 JSON 문서 |
| 최소 권한 원칙 | 필요한 만큼만 권한을 부여, 미리 넓게 주지 않음 |
| MFA / Secrets Manager / Systems Manager | 추가 인증 / 자격증명 안전 보관·자동 교체 / 패치·설정 중앙 관리 |
| IAM Identity Center / 크로스 계정 역할 / 페더레이션 ID | 여러 계정 통합 로그인 / 계정 간 권한 위임 / 외부 자격증명 연동 |
