지금까지 컴퓨팅(처리)과 스토리지(보관)를 다뤘다. 이번엔 이 자원들이 서로 어떻게 연결되고, 외부와는 어떻게 통신하는지를 본다.

## 네트워킹 서비스란

AWS 안에 자원을 만들어도 아무 네트워크에나 막 흩어져 있는 게 아니다. **VPC(Virtual Private Cloud)**는 AWS 안에 내가 경계를 그어서 만드는 **나만의 사설 네트워크**다. 회사 건물 안에 울타리를 치고 그 안에서 어떤 방을 어디에 두고, 어느 출입구로 외부와 연결할지를 내가 설계하는 것과 비슷하다. VPC 안에서 IP 대역을 정하고, 더 작은 구역(서브넷)으로 나누고, 외부 인터넷과의 출입구를 직접 통제한다.

## 핵심 개념과 구조

### Region · AZ · VPC · Subnet 계층 구조

![Region · AZ · VPC · Subnet 계층 구조](/assets/images/aws_region_az_vpc_subnet.png)

1번 글에서 글로벌 인프라를 잠깐 다뤘는데, 여기서 한 단계 더 들어간다.

- **Region(리전)**: 서울, 도쿄처럼 지리적으로 분리된 큰 단위
- **Availability Zone(AZ)**: 한 리전 안에서 물리적으로 분리된 데이터센터 묶음
- **Edge Location**: 사용자와 가장 가까운 캐싱 지점 (CloudFront가 사용)

**VPC는 하나의 리전 전체에 걸쳐 생성**되고, 그 안의 **서브넷은 AZ 하나에만 속한다**. 그림처럼 VPC라는 큰 울타리 안에 AZ-a, AZ-c 같은 구역이 있고, 각 AZ 안에 서브넷이 들어있는 구조다. 같은 VPC 안에서도 서브넷을 여러 AZ에 나눠 만들어두면, 한 AZ에 장애가 나도 다른 AZ의 서브넷이 서비스를 이어받을 수 있어 **고가용성(Multi-AZ)**이 확보된다. 재해 복구처럼 리전 단위 장애까지 대비해야 한다면 다른 리전에도 복제하는 **Multi-Region** 구성을 쓴다.

### VPC 구성요소

- **서브넷(Subnet)**: VPC를 더 작게 나눈 구역. 인터넷과 직접 통신 가능하게 설정하면 **퍼블릭 서브넷**, 외부와 직접 연결되지 않게 하면 **프라이빗 서브넷**이 된다. 웹서버는 퍼블릭에, 데이터베이스는 프라이빗에 두는 식으로 역할을 나눈다
- **인터넷 게이트웨이(Internet Gateway, IGW)**: VPC와 인터넷을 연결하는 출입구. 퍼블릭 서브넷의 리소스가 외부와 통신하려면 IGW가 필요하다
- **NAT 게이트웨이**: 프라이빗 서브넷의 리소스가 (외부에서 들어오는 연결은 막은 채) 외부로 나가는 연결만 할 수 있게 해준다. 예를 들어 프라이빗 서브넷의 서버가 보안 패치를 다운로드할 때 쓴다

### 보안 경계 — Network ACL vs 보안 그룹

둘 다 트래픽을 걸러내지만 적용 범위와 동작 방식이 다르다.

- **네트워크 ACL(NACL)**: **서브넷 단위**로 적용되는 방화벽. 허용/차단 규칙을 순서대로 평가하고, **스테이트리스(stateless)**라서 들어오는 트래픽과 나가는 트래픽 규칙을 각각 따로 정해줘야 한다
- **보안 그룹(Security Group)**: **인스턴스(ENI) 단위**로 적용되는 방화벽. 허용 규칙만 만들 수 있고, **스테이트풀(stateful)**이라서 들어오는 연결을 허용하면 그에 대한 응답은 자동으로 나갈 수 있다

### DNS — Amazon Route 53

도메인 이름(예: example.com)을 IP 주소로 변환해주는 **DNS** 서비스다. 도메인 등록, 헬스 체크, 트래픽을 가장 가까운 리전으로 보내는 라우팅 정책 등을 제공한다.

### 온프레미스 연결

회사 데이터센터와 AWS를 연결하는 방법에는 두 가지가 있다.

- **AWS Site-to-Site VPN**: 인터넷을 통해 암호화된 터널을 만들어 연결한다. 설정이 빠르고 비용이 낮지만, 인터넷 회선 상태에 따라 속도와 안정성이 흔들릴 수 있다
- **AWS Direct Connect**: AWS와 물리적인 전용선으로 직접 연결한다. 안정적이고 빠르지만 구축에 시간이 걸리고 비용이 더 높다
- **AWS Client VPN**: 개별 사용자(노트북 등)가 VPN 클라이언트로 VPC에 접속할 수 있게 해준다 (재택근무자가 사내망처럼 접근할 때)

### 기타 네트워킹 서비스

- **Amazon CloudFront**: Edge Location을 활용하는 **CDN**. 콘텐츠를 사용자와 가까운 곳에 캐싱해서 빠르게 전달한다
- **Amazon API Gateway**: API 요청을 받아 Lambda 등 백엔드로 연결해주는 관문. 인증, 요청 제한(throttling) 등을 처리한다
- **AWS Global Accelerator**: 사용자의 트래픽을 AWS의 글로벌 백본 네트워크를 통해 가장 가까운 엔드포인트로 보내 지연시간을 줄인다
- **AWS PrivateLink**: 인터넷을 거치지 않고 VPC와 AWS 서비스(또는 다른 VPC) 사이를 사설 연결로 묶어준다
- **AWS Transit Gateway**: 여러 VPC와 온프레미스 네트워크를 중앙의 허브 하나로 연결해서, 1:1로 일일이 연결할 필요 없이 네트워크를 단순화한다

## 어디에 쓰이는가

회사 데이터센터를 AWS와 연결할 때, 트래픽이 많지 않거나 빠르게 구축해야 한다면 **VPN**으로 시작한다. 안정적인 대용량 트래픽(예: 데이터베이스 복제, 대규모 마이그레이션)이 꾸준히 필요하다면 **Direct Connect**로 전용선을 구축한다. 둘을 함께 써서 Direct Connect를 기본 경로로, VPN을 장애 시 대체 경로로 구성하는 경우도 많다.

## 직접 해보기 (AWS 콘솔)

VPC 콘솔의 "VPC 마법사(생성)"를 이용하면 퍼블릭 서브넷과 프라이빗 서브넷이 포함된 VPC를 몇 번의 클릭으로 만들어준다. 생성 후 라우팅 테이블을 열어 퍼블릭 서브넷에는 인터넷 게이트웨이로 가는 경로가, 프라이빗 서브넷에는 NAT 게이트웨이로 가는 경로가 각각 어떻게 설정돼 있는지 비교해보자.

## 시험 포인트

- **네트워크 ACL vs 보안 그룹**: NACL은 서브넷 단위·스테이트리스(양방향 규칙 필요), 보안 그룹은 인스턴스 단위·스테이트풀(응답은 자동 허용)
- **VPN vs Direct Connect**: VPN은 인터넷 기반 암호화 터널(빠른 구축, 저비용, 변동성), Direct Connect는 전용선(안정적, 고비용, 구축 시간 필요)
- **AZ vs Region**: AZ는 한 리전 안에서 물리적으로 분리된 데이터센터 묶음, Region은 지리적으로 분리된 큰 단위. AZ는 Region에 속한 하위 개념

| 보안 경계 | 네트워크 ACL / 보안 그룹 | 서브넷 단위·스테이트리스 / 인스턴스 단위·스테이트풀 |
