이전 글에서는 보안에 대해 "누가 무엇을 책임지는가"(공유 책임 모델)와 "누가 무엇을 할 수 있는가"(IAM)를 다뤘다.
경계와 권한을 정했다면, 이제는 "어떤 도구로 실제로 지키고, 무슨 일이 일어났는지 감시하고 증명하는가"로 넘어갈 차례다.

## 보안 거버넌스란

보안이라고 하면 보통 "막는 것"부터 떠올린다. 방화벽을 세우고, 권한을 제한하고, 암호화를 거는 것들이다.
하지만 아무리 잘 막아도 100% 차단은 불가능하고, 사고는 일어난다. 그래서 보안 거버넌스에는 막는 것 외에 두 가지가 더 필요하다.

**감시(monitoring/detection)**는 이상한 일이 일어나고 있는지 계속 지켜보는 것이다. 누군가 비정상적인 위치에서 로그인을 시도하거나, 알려진 악성 IP와 통신하는 EC2 인스턴스가 있는지 자동으로 탐지한다.
**증명(audit/compliance)**은 "우리는 규정을 지키고 있다"는 것을 기록과 보고서로 보여주는 것이다. 누가 언제 무엇을 했는지 기록이 남아있어야 사고가 났을 때 원인을 추적하고, 외부 감사를 받을 때도 증빙할 수 있다.

즉 보안 거버넌스는 "막기 + 감시하기 + 증명하기"의 조합이고, AWS는 이 세 가지에 각각 대응하는 서비스군을 제공한다.

## 핵심 개념과 구조

### 탐지 → 통합 관제 → 대응 흐름

![AWS 보안 탐지·대응 흐름](/assets/images/aws_threat_detection_flow.png)

위협을 다루는 흐름은 보통 이렇게 진행된다. 여러 탐지 서비스가 각자 맡은 영역을 감시하고, 그 결과가 한곳에 모이고, 모인 정보를 바탕으로 대응이 이루어진다.

**위협 탐지** 단계에는 역할이 겹치지 않는 4개 서비스가 있다.

- **Amazon GuardDuty**: VPC 흐름 로그, DNS 로그, CloudTrail 로그를 분석해서 "이상행동"을 탐지한다. 예를 들어 평소와 다른 국가에서 API 호출이 들어오거나, 암호화폐 채굴처럼 보이는 트래픽이 발생하면 알려준다
- **Amazon Inspector**: EC2와 컨테이너 이미지, Lambda 함수를 스캔해서 알려진 **취약점(CVE)**이나 보안 모범사례 위반이 있는지 찾아낸다. GuardDuty는 "지금 일어나고 있는 이상행동"을, Inspector는 "잠재적으로 뚫릴 구멍"을 본다는 점이 다르다
- **Amazon Macie**: S3에 저장된 데이터를 스캔해서 주민등록번호, 신용카드 번호 같은 **민감정보(PII)**가 노출된 채로 저장돼 있는지 찾아낸다
- **Amazon Detective**: GuardDuty 등에서 탐지가 발생했을 때, 관련된 로그를 모아 시각화해서 "이 사건이 정확히 어떻게 일어났는지" 원인 분석을 돕는다

**통합 관제**는 **AWS Security Hub**가 담당한다. GuardDuty, Inspector, Macie를 비롯한 여러 보안 서비스의 알림을 한 화면에 모아서 보여주고, 보안 표준(CIS 벤치마크 등) 준수 여부도 점수로 보여준다. 서비스마다 따로 콘솔을 열어볼 필요 없이 Security Hub 하나만 보면 된다.

**대응**은 EventBridge로 특정 탐지 이벤트를 받아 Lambda 함수를 자동으로 실행해 조치하거나(예: 의심스러운 보안 그룹 규칙을 자동으로 되돌리기), SNS로 담당자에게 알림을 보내는 방식으로 이루어진다.

### 네트워크 보호

- **AWS WAF(Web Application Firewall)**: SQL 인젝션, XSS 같은 **애플리케이션 계층(L7)** 공격을 막는다. 웹 요청의 내용을 보고 규칙에 맞으면 차단한다
- **AWS Shield**: **DDoS(분산 서비스 거부) 공격**을 막는다. 모든 계정에 기본으로 적용되는 Standard와, 더 강력한 보호와 전담 대응팀 지원을 받는 유료 Advanced가 있다
- **AWS Firewall Manager**: 여러 계정에 흩어진 WAF 규칙, Shield 설정, 보안 그룹을 한곳에서 중앙 관리한다

### 암호화

데이터는 두 가지 상태에서 보호되어야 한다. **저장 중 암호화(encryption at rest)**는 디스크에 저장된 데이터를 암호화하는 것이고, **전송 중 암호화(encryption in transit)**는 네트워크로 이동하는 동안의 데이터를 암호화하는 것(TLS/HTTPS)이다. 둘 다 챙겨야 한 쪽만 뚫려도 데이터가 안전하다.

- **AWS KMS(Key Management Service)**: 암호화에 쓰는 키를 생성하고 관리한다. S3, EBS, RDS 등 대부분의 AWS 서비스가 KMS와 연동된다
- **AWS CloudHSM**: 전용 하드웨어 보안 모듈(HSM)에서 키를 관리한다. KMS보다 더 엄격한 규제(금융권 등)를 충족해야 할 때 쓴다
- **AWS Certificate Manager(ACM)**: TLS/SSL 인증서를 발급하고 자동으로 갱신해준다. 인증서 만료를 까먹어서 사이트가 멈추는 사고를 막아준다

### 모니터링과 감사

여기서부터는 "증명"에 해당하는 영역이다. 각 서비스가 보는 대상이 다르다는 점이 시험에서 자주 헷갈리는 부분이다.

- **Amazon CloudWatch**: 리소스의 **성능과 상태**를 모니터링한다 (CPU 사용률, 메모리, 로그, 알람). "지금 시스템이 건강한가"를 본다
- **AWS CloudTrail**: AWS 계정에서 **누가 어떤 API를 호출했는지** 기록한다. "누가 무엇을 했는가"를 본다. 콘솔 클릭이든 CLI 명령이든 거의 모든 동작이 API 호출이기 때문에 사실상 모든 행동의 감사 기록이 된다
- **AWS Config**: 리소스의 **설정이 시간에 따라 어떻게 변했는지** 추적한다. "이 보안 그룹이 언제부터 0.0.0.0/0을 허용하게 바뀌었는가"를 본다
- **AWS Audit Manager**: CloudTrail, Config 등에서 모은 증거를 규제 프레임워크(PCI-DSS, GDPR 등) 기준에 맞춰 자동으로 정리해서 **컴플라이언스 증빙 보고서**를 만들어준다

### 컴플라이언스와 운영 점검

- **AWS Artifact**: AWS의 보안/컴플라이언스 감사 보고서(SOC, PCI 등)와 계약서를 다운로드할 수 있는 셀프서비스 포털이다. 외부 감사 대응 시 AWS 쪽 증빙 자료가 필요할 때 여기서 받는다
- **AWS Trusted Advisor**: 보안뿐 아니라 비용, 성능, 내고장성 측면에서 계정을 점검하고 개선 권고를 준다 (예: "이 S3 버킷이 퍼블릭으로 열려 있습니다")
- **AWS Marketplace**: 서드파티 보안 제품(방화벽, 백신, SIEM 등)을 AWS 청구서에 통합해서 구매할 수 있는 장터다
- 지원이 필요할 땐 **AWS Knowledge Center**(자주 묻는 질문 모음)와 **AWS Security Blog**(보안 사례와 모범 사례 공유)를 참고할 수 있다

## 어디에 쓰이는가

실무에서는 보통 GuardDuty와 Security Hub를 항상 켜두고, 알림이 뜨면 보안팀이 Security Hub에서 심각도를 확인한다.
심각도가 높으면 Detective로 들어가 관련 로그를 모아 원인을 분석하고, 사람이 직접 조치하거나 EventBridge+Lambda로 미리 정해둔 자동 대응(예: 침해된 IAM 자격증명 비활성화)을 실행한다.
분기마다 외부 감사를 받을 때는 Audit Manager로 모아둔 증거와 Artifact에서 받은 AWS 측 보고서를 함께 제출한다.

## 직접 해보기 (AWS 콘솔)

GuardDuty 콘솔에서 "지금 활성화" 버튼 하나만 누르면 별도 설정 없이 바로 탐지가 시작된다 (30일 무료 평가판 제공).
CloudTrail 콘솔의 "이벤트 기록(Event history)"에서는 최근 90일간 내 계정에서 일어난 API 호출을 검색해볼 수 있다. 예를 들어 본인이 로그인한 시각이나 EC2 인스턴스를 생성한 기록을 직접 찾아보면 "누가 무엇을 했는지"가 어떻게 남는지 감이 온다.

## 시험 포인트

- **GuardDuty vs Inspector vs Macie**: GuardDuty는 실시간 이상행동 탐지, Inspector는 알려진 취약점 스캔, Macie는 S3 안의 민감정보 탐지. 셋은 "무엇을 보는가"가 다르므로 역할이 겹치지 않는다
- **CloudTrail vs CloudWatch vs Config**: CloudTrail은 "누가 무엇을 했는지"(API 호출 기록), CloudWatch는 "지금 상태가 어떤지"(성능/상태 모니터링), Config는 "설정이 어떻게 바뀌었는지"(변경 이력 추적)
- **WAF vs Shield**: WAF는 애플리케이션 계층(L7) 공격(SQL 인젝션 등) 방어, Shield는 DDoS 공격 방어. 공격의 종류가 다르다

## 정리

| 구분 | 서비스 | 역할 |
|---|---|---|
| 위협 탐지 | GuardDuty / Inspector / Macie / Detective | 이상행동 / 취약점 / 민감정보 탐지 / 원인 분석 |
| 통합 관제 | Security Hub | 여러 보안 서비스 알림을 한 화면에 모음 |
| 네트워크 보호 | WAF / Shield / Firewall Manager | 애플리케이션 공격 방어 / DDoS 방어 / 규칙 중앙 관리 |
| 암호화 | KMS / CloudHSM / ACM | 키 관리 / 전용 하드웨어 키 관리 / TLS 인증서 발급·갱신 |
| 모니터링·감사 | CloudWatch / CloudTrail / Config / Audit Manager | 성능 모니터링 / API 호출 기록 / 설정 변경 추적 / 컴플라이언스 증빙 |
| 컴플라이언스·점검 | Artifact / Trusted Advisor / Marketplace | 감사 보고서 다운로드 / 운영 점검 / 서드파티 보안 제품 |
